# MindLeak v0.1.2 independent-developer pilot

## Purpose

This pilot tests external adoption, not just engineering correctness or a
controlled benchmark. It asks whether independent developers can install the
public release without live setup assistance, reach useful context quickly, and
keep using it during real multi-agent work for seven days.

Recruit 3-5 developers who already use more than one coding agent. At least two
must complete the full seven-day observation window before any retention or
external-adoption result can be reported.

Recruitment and enrollment status: https://github.com/monk-eee/MindLeak/issues/8

## Eligibility and consent

A participant must:

- be independent of the MindLeak implementation team;
- already use multiple coding agents in normal development work;
- use a real repository they are authorized to work in;
- install only from the public v0.1.2 release assets;
- consent to the bounded observations below; and
- be free to stop, disable, or remove MindLeak at any time without explanation.

Use an anonymous participant id such as `p01`. Do not publish names, employer
names, repository names, source, prompts, model reasoning, tokens, terminal
output, database contents, credentials, or unredacted screenshots. Free-text
feedback must be reviewed with the participant before publication.

## No-assistance boundary

The install attempt starts from the public release page:

https://github.com/monk-eee/MindLeak/releases/tag/v0.1.2

The participant verifies `SHA256SUMS`, follows public documentation, and records
what happens without synchronous maintainer help. Clarifying a study question is
allowed; diagnosing or fixing the installation live is not. A failed install is
a result and stays in the aggregate.

After the independent attempt is recorded, a participant may use public issues
or documentation for later troubleshooting. Record that separately.

## Observation schedule

### Day 0: install and first value

Record:

- anonymous participant id;
- operating system, architecture, editor/CLI client, and coding-agent clients;
- exact asset name, checksum result, and both MCP `serverInfo.version` values;
- install start/end timestamps and whether registration succeeded;
- time from first activation to the first useful graph, impact, or recall result;
- whether that result was useful, irrelevant, empty, or unavailable;
- optional/degraded capabilities reported by the product;
- whether two concurrent sessions received distinct attribution/claim owners;
- duplicate work, collision, or unexpected ownership behavior; and
- any disable/removal event and reason volunteered by the participant.

The first-use target is a useful graph or impact result within five minutes. A
miss remains evidence; do not coach the participant until the attempt is closed.

### Day 1

Record whether MindLeak was used again, which client path was used, one useful
and one irrelevant/missing context example when available, resumed-task success,
session attribution behavior, collisions/duplicates, degraded capabilities, and
whether the participant disabled or removed it.

### Day 7

Record days used, whether it was used on real work on day 7, useful versus
irrelevant context counts or bounded examples, resumed-task success, distinct
session ownership, duplicate/collision observations, degraded capabilities, and
retained/disabled/removed status with an optional reason.

## Aggregation and publication

Publish only anonymized aggregate counts and bounded, participant-reviewed
examples in `docs/EVALUATION.md` and a dated JSON artifact under
`benchmarks/results/`. Include every install failure, dropout, disable/removal,
degraded capability, duplicate/collision, and irrelevant-context report.

At minimum report:

- recruited, installed, day-1, and day-7 participant counts;
- median/range install and first-useful-context time;
- successful distinct-session and resumed-task observations;
- duplicate/collision and degraded-capability counts;
- retained/disabled/removed counts and volunteered reasons; and
- platforms, clients, build identities, limitations, and missing data.

Do not broaden efficacy, adoption, or retention claims unless the observed cohort
supports them. A small or incomplete cohort is reported as such, not rounded up
to a success.
